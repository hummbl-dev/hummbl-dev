# How to Participate in Discussions

HUMMBL uses GitHub Discussions as the primary surface for public conversation before PRs. Discussions are open on selected repos — see the list below.

## Where Discussions are enabled

| Repo | Discussion focus |
|---|---|
| [hummbl-dev/hummbl-dev](https://github.com/hummbl-dev/hummbl-dev/discussions) | General questions, fleet navigation, contributor onboarding |
| [hummbl-governance](https://github.com/hummbl-dev/hummbl-governance/discussions) | Governance primitive design, feature requests |
| [base120](https://github.com/hummbl-dev/base120/discussions) | Mental model design, framework questions |
| [arbiter](https://github.com/hummbl-dev/arbiter/discussions) | Code quality scoring, governance compliance |
| [hummbl-agent](https://github.com/hummbl-dev/hummbl-agent/discussions) | Agent orchestration patterns |
| [agent-handoffs](https://github.com/hummbl-dev/agent-handoffs/discussions) | Context transfer, delegation patterns |
| [claim-evidence-ledger](https://github.com/hummbl-dev/claim-evidence-ledger/discussions) | Evidence schemas, provenance chains |
| [execution-receipts](https://github.com/hummbl-dev/execution-receipts/discussions) | Receipt formats, audit trails |
| [agent-runtime-governance](https://github.com/hummbl-dev/agent-runtime-governance/discussions) | Kill switches, circuit breakers, guardrails |
| [packages](https://github.com/hummbl-dev/packages/discussions) | Package distribution, release pipeline |

## Discussion norms

1. **Be specific.** Reference repos, issues, or docs when possible.
2. **Distinguish canon from candidate.** Most repos are `v0.1-packet` (non-canon). Don't treat exploratory content as adopted standard.
3. **No canon proposals in Discussions.** If you want to propose something for canonical adoption, open an issue with a receipt.
4. **Receipts over vibes.** If you claim something, link the evidence.
5. **Respect operator authority.** Some decisions require the operator. Don't push for autonomous resolution of decision-required issues.
6. **Be appreciative and precise.** This is a governed ecosystem, not a debate stage.

## What belongs in Discussions vs Issues

| Surface | Use for |
|---|---|
| Discussions | Questions, ideas, exploration, feedback, design discussion |
| Issues | Specific bugs, feature requests, docs corrections, starter tasks |
| PRs | Code, docs, schema, or fixture changes with receipts |

When in doubt, start in Discussions. A maintainer will route you to Issues or PRs if needed.
