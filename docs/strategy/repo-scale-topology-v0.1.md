## Purpose

Define which HUMMBL repositories should intentionally become large, what kind of scale is legitimate for each repository class, and when repository growth indicates healthy accumulation versus responsibility collapse, generated-artifact debt, or a needed split.

Repository importance must not be inferred from GitHub's reported byte size alone.

## Proposed scale topology

### Largest executable systems

- `founder-mode` / future Ownward control plane
- `hummbl-production`

Legitimate scale drivers: product surfaces, orchestration, adapters, identity, delegation, observability, safety, deployment, and broad test suites.

### Largest knowledge estate

- `hummbl-bibliography`
- `corpus`
- `claim-evidence-ledger` or its admitted canonical successor

Legitimate scale drivers: versioned source records, identities, provenance, corrections/retractions, manifests, claim-source relations, and data-release metadata. Raw large objects should normally live outside Git with hashes and manifests in Git.

### Largest assurance estate

- `hummbl-governance`

Legitimate scale drivers: standards, schemas, validators, threat models, valid/invalid/adversarial fixtures, conformance tests, and compatibility machinery.

### Largest research staging estate

- `hummbl-research`

Legitimate scale drivers: campaigns, methods, source manifests, experimental records, negative results, and reproducibility packets.

### Moderate reusable runtimes

- `hummbl-agent`
- `autoresearch-pipeline`
- `arbiter`
- `mcp-server`

These should remain bounded by interface and runtime responsibility.

### Small, high-authority cores

- `hummbl-doctrine`
- `baseN` / `base120`
- `krineia`
- `hummbl-bus`
- `fleet-standard`
- other protocol/specification kernels

Smallness here improves reviewability, authority clarity, and assurance.

## Required classifications

For each active HUMMBL-native repo, choose one primary disposition:

| Classification | Description |
|----------------|-------------|
| `HEALTHY_SCALE` | Scale is justified by role and responsibility |
| `UNDERBUILT_FOR_ROLE` | Repository needs more content to fulfill its role |
| `BLOATED_BY_GENERATED_OR_BINARY_CONTENT` | Generated/binary content dominates |
| `RESPONSIBILITY_COLLAPSE` | Too many unrelated responsibilities in one repo |
| `SPLIT_CANDIDATE` | Should be split into bounded repos |
| `CONSOLIDATION_CANDIDATE` | Should be merged with another repo |
| `DATA_EXTERNALIZATION_CANDIDATE` | Raw data should move out of Git |
| `ARCHIVE_OR_MIRROR_ONLY` | Should be archived or kept as mirror |
| `MEASUREMENT_INSUFFICIENT` | Cannot classify without more measurement |

## Split triggers

Investigate a split when several of these are present:

- unrelated release cadences
- distinct authority or privacy boundaries
- incompatible dependency stacks
- one subsystem changes much faster than the rest
- separate deployment lifecycles
- separate user populations
- source-of-truth ambiguity
- reviewers cannot meaningfully inspect the whole change surface
- generated/binary content dominates Git
- a repository acts as a convenience sink
- a child subsystem has durable identity, interface, executor relationship, and receipt path

Do not split solely because a repo is large.

## Knowledge-storage rule

For large corpora and intelligence sources, Git should generally preserve:

- manifests
- immutable identifiers and hashes
- source and retrieval metadata
- licensing and access posture
- transformation recipes
- sample fixtures
- claim/evidence relationships
- release and correction receipts

Raw PDFs, OCR corpora, audio, video, embeddings, and large indexes should use appropriate object/data storage unless a bounded review shows Git is the correct medium.
