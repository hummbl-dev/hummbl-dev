# Agent Mutation Guardrails for High-Throughput Agentic Work

**Status: PUBLIC-SAFE PROTOCOL DOCUMENT — VENDOR-NEUTRAL**

Issue: hummbl-dev/hummbl-dev#131
Parent: hummbl-dev/hummbl-dev#125

## Purpose

Document agent mutation guardrails so agents cannot make code appear correct by destructive git operations, hidden stubs, skipped tests, broad resets, or unreviewed scope expansion.

This is directly usable as a preflight checklist for agents before modifying a repo.

## Git/worktree operations

### Hard bans (never execute without explicit operator authorization AND receipt)

- `git push --force` / `git push -f` — rewrites remote history
- `git push --force-with-lease` — still rewrites history; needs approval
- `git reset --hard` to a distant ref — discards uncommitted work
- `git clean -fdx` — deletes untracked files including stashes and work
- `git rebase` on shared branches — rewrites history others depend on
- `git filter-branch` / `git filter-repo` — rewrites all history
- `git reflog expire --expire=now --all && git gc --prune=now` — destroys recovery path
- `git branch -D` on branches not created by this agent — deletes others' work
- `git stash drop` / `git stash clear` — destroys stashed work
- `rm -rf` on any directory not created by this agent in this session
- `git checkout .` / `git restore .` — discards all uncommitted changes
- `git commit --amend` on already-pushed commits — rewrites history

### Needs operator approval (not banned, but require explicit authorization)

- `git rebase` on agent-owned feature branches — acceptable if branch is not shared
- `git merge --squash` — acceptable for cleaning up agent's own branch
- `git cherry-pick` across branches — acceptable with scope verification
- `git tag` (any) — acceptable if tag follows repo convention
- `git branch -D` on agent-owned branches — acceptable if branch is fully merged
- `git stash` — acceptable but must be tracked and restored or explicitly parked

### Allowed without approval

- `git add` / `git commit` on agent-owned feature branch
- `git checkout -b` to create new branches
- `git status` / `git diff` / `git log` — read-only operations
- `git fetch` — read-only from remote
- `git pull --rebase` on agent-owned feature branch

## Scope discipline

- **One issue → one bounded PR** unless operator explicitly approves batching
- Do not expand scope beyond the issue's acceptance criteria
- Do not fix unrelated issues encountered during work — file separate issues
- Do not refactor code outside the change's direct dependency path
- Do not update docs, configs, or CI unrelated to the issue
- If a change requires touching multiple repos, document the cross-repo scope and get approval

## Test discipline

### Hard bans

- **Do not delete tests** to make validation pass
- **Do not skip tests** (pytest.mark.skip, @Ignore, xfail without investigation) to make validation pass
- **Do not downscope tests** (reducing assertions, loosening thresholds) to make validation pass
- **Do not comment out test code** to make validation pass
- **Do not change test expectations to match broken code** (adjusting expected values to match bugs)

### Allowed with explicit review

- Mark a test as `xfail` with a linked issue explaining the known failure
- Skip tests that require unavailable infrastructure (with `@pytest.mark.skipif` and clear condition)
- Delete tests that are genuinely obsolete (with explanation in commit message and PR description)
- Adjust test thresholds when the underlying contract has changed (with contract reference)

## Stub discipline

### Hard bans

- **No placeholder implementations presented as done**
- **No TODO-driven fake passes** (code that returns hardcoded expected values)
- **No panic/throw placeholders** (`raise NotImplementedError` in production paths presented as complete)
- **No `pass`-only function bodies** presented as implementations
- **No mock implementations in non-test code** presented as real implementations

### Allowed with explicit labeling

- Stub implementations marked with `# STUB: not yet implemented — tracked in #NNN`
- Placeholder returns marked with `# TODO: implement in follow-up — tracked in #NNN`
- `NotImplementedError` in abstract base classes or interface definitions (not in concrete implementations)

## Generated-code discipline

- Generated files must state: source tool, generation command, and review status
- Generated files must include a header comment: `# Generated by <tool> on <date>. Do not edit manually.`
- Regeneration commands must be documented in the repo or PR description
- Generated files must be reviewed for correctness, not just committed blindly
- Large generated files should be committed separately from hand-written changes

## Command budget

- Avoid slow or global commands unless scoped/approved for the repo
- Do not run `grep -r` / `rg` across the entire filesystem
- Do not run `find /` or similar system-wide searches
- Scope searches to the repo directory
- Do not run `npm install` / `pip install` without checking for existing lockfiles
- Do not run global config changes (`git config --global`, `npm config set`)
- Do not run network operations without verifying the target is expected

## Receipt discipline

For every mutation session, record:

- Commands run (exact command lines)
- Files changed (added, modified, deleted)
- Tests run and results
- Failures observed and how they were addressed
- Git operations performed
- Scope of changes (files, lines, affected modules)
- Any deviations from the issue's acceptance criteria
- Any operator approvals received (with timestamp and scope)

## PR self-check

Paste this into PR descriptions before requesting review:

```markdown
## Agent mutation self-check

- [ ] All changes are within the issue's acceptance criteria scope
- [ ] No destructive git operations performed (or explicitly approved and receipted)
- [ ] No tests deleted, skipped, or downscoped to pass validation
- [ ] No stubs, placeholders, or TODO-driven fake passes presented as done
- [ ] Generated files (if any) are labeled with source, command, and review status
- [ ] No unrelated files modified (docs, configs, CI outside scope)
- [ ] All commands run are documented in the PR description
- [ ] No global config changes or system-wide operations performed
- [ ] Receipt of commands, failures, and changes is included
```

## Adversarial-review checklist

For reviewers detecting hidden stubs or skipped tests:

- [ ] Search for `TODO`, `FIXME`, `STUB`, `NotImplementedError`, `pass` in changed files
- [ ] Search for `skip`, `xfail`, `@Ignore`, `pytest.mark.skip` in test files
- [ ] Compare test count before and after the change (should not decrease without explanation)
- [ ] Check for hardcoded return values that match test expectations
- [ ] Check for commented-out test code
- [ ] Check for test assertion changes that loosen thresholds
- [ ] Verify generated files have proper headers and regeneration commands
- [ ] Check git log for any force-pushes, rewrites, or history manipulation
- [ ] Verify no unrelated files were modified
- [ ] Check for mock implementations in non-test code

## Non-goals

- No blanket prohibition on legitimate git operations
- No replacement for branch protection or CI
- No private local-machine assumptions
- No repo-wide enforcement code (separately scoped if needed)
