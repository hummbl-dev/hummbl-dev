# 37,000 Lines of Slop: A Full Analysis

> A breakdown of Scott's video essay critiquing Gary Tan's AI coding claims — and what it reveals about the state of AI-assisted development in 2026.

---

## Context

Gary Tan, CEO of Y Combinator and creator of **GStack** (a system of markdown "skill files" that prompt AI agents to write code), bragged on Twitter about writing 37,000 lines of code per day across five projects, sustained over a 72-day shipping streak.

A user named Ishwar Jha (@IshwarJha on X) audited Gary's production blog — one of those projects — and published the results. What followed was a case study in the gap between AI-generated output volume and actual software quality.

Scott's video essay uses this as a jumping-off point for a broader argument about AI psychosis in coding, who to trust for advice, and what responsible AI-assisted development actually looks like.

---

## Insight 1: Lines of Code is a Broken Metric — and Bragging About It is the Tell

**The claim:** 37,000 lines/day = productivity, shipping velocity, and proof of value.

**The reality:** Lines of code (LOC) is one of the oldest discredited productivity proxies in software engineering. It was rejected by serious practitioners decades ago. The fact that it's making a comeback as an AI flex exposes how many people entering AI-assisted development have no baseline in software engineering epistemology.

**Why it's worse with AI:** When a human writes 37,000 lines, there's at least implied comprehension — they typed it, they thought it. When AI writes 37,000 lines, the author's role is prompt author and (at best) reviewer. At 37K lines/day, no human can do meaningful review. The metric has been fully decoupled from understanding.

**The signal inversion:** Bragging about LOC generated by AI is the equivalent of a chef bragging about how many ingredients they bought. Acquisition is not cooking. Generation is not engineering.

**What should be tracked instead:**
- Lines reviewed and understood per day (the real throughput constraint)
- Test coverage on generated code
- Defect density in shipped code
- Reduction in technical debt over time
- Architectural coherence across the codebase

**Bottom line:** 37K lines/day isn't a flex. It's a warning that nobody is reading the output.

---

## Insight 2: The Democratization Double-Edge

**The upside:** AI has genuinely enabled people without engineering backgrounds to build things they never could have. This is real. The barrier to entry for creating functional software has dropped dramatically.

**The downside:** The same capability gap that prevented non-engineers from building also prevented them from shipping broken things at scale. That gate is now open in both directions. People who don't know what a production bundle is, what alt tags are for, or why you compress images before serving them are now shipping to production daily.

**The nuance:** This isn't an argument against democratization. It's an argument that democratization without comprehension creates a new category of risk — specifically, confident ignorance at scale. The person who doesn't know they're doing something wrong is far more dangerous than the person who knows they're cutting corners.

**The structural problem:** Traditional gatekeeping mechanisms (code review, senior engineers, QA pipelines) were downstream filters on bad code. When the person generating the code is also the one declaring it done, those filters are bypassed entirely. GStack had a QA skill — a markdown file prompting AI to review its own output — and it still shipped 300KB of test files to production. AI reviewing AI output, without a human who understands both, is not quality assurance.

---

## Insight 3: What "Slop" Actually Looks Like in Production

The audit of Gary's blog is instructive precisely because the failures are not exotic. They are embarrassingly basic:

### 300KB of Test Files in the Production Bundle
Test files belong in the development environment. Shipping them to every visitor is a fundamental build pipeline failure — the kind of thing that a one-line `.gitignore` or a basic build configuration check would catch. This isn't a subtle bug. It's a sign that nobody audited what the build output contained.

**Implication:** The AI generated tests (arguably good), but no one understood the build system well enough to ensure they stayed out of the production artifact.

### Uncompressed Multi-Megabyte PNGs
A 2MB PNG served to every visitor, uncompressed. Every major frontend framework (Next.js, Astro, Nuxt, SvelteKit, etc.) ships with automatic image optimization. This failure requires actively not using the tooling, or using AI-generated code that doesn't engage with it.

**Implication:** The AI scaffolded something that worked locally without triggering the image pipeline. No one noticed because no one was looking at the network tab.

### Backend Tooling Bleeding into the Frontend Bundle
A 520KB WYSIWYG rich text editor (Trix, a Rails ecosystem tool) ending up in the frontend bundle. The auditor speculates this is a spillover from the backend framework. The mechanism of how this happened is almost less important than the fact that it wasn't caught — because it requires looking at bundle composition to notice.

**Implication:** AI can generate code that technically functions while having deeply confused architecture. Without understanding the system boundaries, the generated code can blur them invisibly.

### Entire DOM Rendered Twice
The page content was rendered once for mobile and once for desktop, both present in the DOM simultaneously. This is a 2003-era CSS problem that responsive design frameworks solved. It doubles DOM size, increases parse time, and degrades performance. It also suggests the AI was given conflicting or poorly scoped instructions about responsive layout.

**Implication:** AI solves problems locally — it may have been told "make this work on mobile" and generated a parallel DOM tree rather than refactoring the existing layout. Classic local solution to a global problem.

### 47 Images with No Alt Tags
Accessibility is not optional. It's increasingly a legal requirement (ADA, EU Accessibility Act). Having a QA "skill" that doesn't catch 47 missing alt tags suggests the QA prompt either didn't include accessibility criteria or the AI reviewer didn't know what to look for.

**Implication:** Skills/prompts only encode what their author thinks to encode. If the author doesn't know about accessibility, the skill won't check for it. Comprehension gaps in the author propagate into the skill.

### Lighthouse Score of 80 Despite the Above
This is perhaps the most important detail. Automated performance scoring masked the structural failures. An 80 is a "good" score by most interpretations. Anyone who ran Lighthouse and called it done would conclude the site was fine.

**Implication:** Automated validation creates false confidence. Lighthouse measures a narrow slice of quality — it doesn't check bundle composition, accessibility semantics at depth, DOM correctness, or architectural coherence. Treating it as a quality gate is itself a comprehension failure.

## Insight 4: Prompt-Engineering Your Way to Quality Doesn't Work

GStack is, at its core, a collection of markdown files that instruct AI agents to write good code. There's a QA skill. There's presumably a code review skill. The system promises that skills → quality.

**The failure mode:** Skills encode the author's existing knowledge, vocabulary, and blind spots. They can't encode what the author doesn't know. If the author doesn't know that test files shouldn't ship to production, there's no skill that will check for it. If the author doesn't know about alt tags, the accessibility skill doesn't get written.

**The deeper problem:** "Make it good, no bugs" is not a specification. It's a hope. AI systems optimize for satisfying the prompt, not for satisfying unstated quality criteria the prompter didn't know to include. The gap between what you asked for and what you actually need is the comprehension gap — and no skill file bridges it automatically.

**The meta-irony:** Gary's GStack system was itself probably built with GStack. If the author's comprehension gap is baked into the tool, and the tool is used to build itself, the gap compounds rather than closes. Meta-recursive slop generation.

---

## Insight 5: AI Solves Problems Locally, Not Globally

This is stated explicitly in the video and is one of the most practically important observations.

**The pattern:** When AI encounters a constraint or problem, it tends to solve it at the nearest available scope. A mobile layout problem gets solved by adding a parallel DOM tree, not by refactoring the layout system. A missing dependency gets solved by adding it locally, not by evaluating whether the architecture needs it at all.

**Why this happens:** AI models are trained on code that solves problems. Most training examples are local fixes. Global architectural refactors are rare, high-context operations that are underrepresented in training data and require sustained reasoning over large contexts.

**The accumulation effect:** Over time, local solutions accumulate into a codebase with high complexity, high duplication, and no coherent global structure. The codebase "works" in that individual features function, but it's unmaintainable, unpredictable, and increasingly expensive to extend. This is the natural endpoint of unconstrained AI generation without architectural oversight.

**The countermeasure:** Static analysis tools that detect duplication, dead code, and complexity hotspots (like "fallow," mentioned in the video) are specifically valuable because they make the accumulation visible. AI won't surface this on its own — it takes a deterministic tool that doesn't care about whether the code works, only about its structure.

## Insight 6: The Accountability Gap and the New Class of Grifters

**The structural dynamic:** AI has lowered the floor for building things and raised the ceiling for what any individual can ship. It has not, however, created accountability for what gets shipped. The people with the largest audiences in the AI coding space are frequently optimizing for content virality (streaks, LOC counts, "I built X in 2 hours") rather than for engineering quality.

**The grifter archetype:** An "AI coding coach" who doesn't understand production systems is actively harmful when they have a large audience. They normalize bad practices, create aspirational models based on vanity metrics, and occupy the authority position in communities of learners who don't yet know enough to evaluate the advice. Gary Tan has significant credibility as YC CEO — his bragging sets a norm.

**The information asymmetry:** Beginners in AI-assisted development can't easily distinguish "person who shipped a lot" from "person who shipped good things." LOC counts and streak lengths are legible. Code quality is not, especially if you can't read the code.

**What accountability would look like:**
- Showing audits of your own production code, not just the LOC count
- Measuring defect density, not just shipping velocity
- Publishing post-mortems when AI-generated code causes failures
- Being explicit about what was reviewed vs. what was trusted blindly

## Insight 7: Deterministic Tools Fill the Gap AI Can't

**The core argument:** AI for generation + deterministic tools for validation is the correct architecture for AI-assisted development.

**Why AI can't fully validate its own output:**
- AI reviews are probabilistic — the same model that generated the bug may not catch it
- AI doesn't maintain a global model of the codebase across sessions
- AI optimizes for satisfying review prompts, not for finding problems
- "Look for bugs" finds bugs the reviewer knows to look for

**What deterministic tools actually do:**
- Dead code detection: finds code that exists but is never called (accumulation artifact)
- Duplication detection: finds the same problem solved locally N times
- Complexity measurement: identifies hotspots where cognitive load is highest and bugs are most likely
- Bundle analysis: surfaces what's actually in production artifacts (test files, oversized dependencies)
- Accessibility linting: catches missing alt tags, ARIA issues, contrast failures mechanically

**The key property:** Deterministic tools are indifferent to whether the code "works." They measure structure, not function. This is the dimension AI generation most consistently neglects.

**The workflow implication:** Run static analysis as a mandatory gate, not an optional audit. The output of AI generation should go through deterministic validation before any human review — it's a cheap first filter that surfaces structural problems mechanically.

## Insight 8: The Correct Mental Model for AI-Assisted Development

Mario Zechner's framework (cited in the video) and Scott's own practice converge on the same model:

### The Zechner Model
1. Identify work that is rote, boring, or well-specified (high AI-appropriateness)
2. Let AI generate it
3. Evaluate the output with comprehension — read it, understand it
4. Finalize implementation (AI-assisted or manually) after evaluation
5. Set a self-imposed daily limit on AI-generated code volume calibrated to your review capacity

**The key constraint:** Your review capacity is the real throughput bottleneck, not AI generation speed. Optimizing generation speed beyond your review capacity produces unreviewed code in production. This is the 37K lines/day problem.

### The "Use Your Brain to Outsource Work" Distinction

This is the most quotable formulation and the most important conceptual boundary:

- **Wrong:** Outsource your brain (let AI decide what to build, how to build it, whether it's good)
- **Right:** Use your brain to outsource work (you understand the system, you specify precisely, AI executes the rote parts, you evaluate and own the result)

The difference is comprehension ownership. In the right model, you could explain every component of your system, why it's structured the way it is, and what the tradeoffs are. In the wrong model, you couldn't — and you wouldn't know you couldn't until it fails in production.

### The Slowing Down Principle
Speed is not the constraint AI-assisted development should optimize for. The marginal cost of generation is near zero. The marginal cost of reviewing, understanding, and integrating is human time and attention. Structuring your workflow around generation speed is optimizing the wrong variable.

## Insight 9: Governance is the Unspoken Requirement

The video doesn't use the word "governance," but the failures it describes are governance failures:

- **No audit trail:** Nobody checked what was in the production bundle
- **No policy enforcement:** There was no rule preventing test files from shipping
- **No ground truth:** Nobody had a canonical understanding of what "done" and "good" meant
- **No accountability loop:** The 72-day streak was the metric, not the quality of what shipped
- **False validation:** Lighthouse and AI QA skills created the appearance of review without the substance

**The governance gap in AI-native teams:** As teams adopt AI-generated code at scale, the traditional mechanisms that enforced quality (human code review, senior engineer judgment, QA pipelines) are being bypassed in the name of speed. What replaces them is usually nothing — or cosmetic substitutes like AI self-review.

**What actual governance looks like:**
- Policy-as-code: machine-enforceable rules about what can ship (no test files in bundles, all images must have alt tags, bundle size limits)
- Audit trails: a log of what was generated, by what model, reviewed by whom
- Agent accountability: if an AI agent generates the code, the policy system should be able to attribute failures
- Human-in-the-loop gates: defined checkpoints where human comprehension is required before proceeding
- Separation of generation and validation: the agent that generates should not be the sole validator

**The GStack failure as a governance case study:** GStack is a generation system with a cosmetic validation layer (the QA skill). It has no enforcement, no audit, no ground truth outside the author's intent. When the author's intent is "ship fast," that's what the system optimizes for — regardless of what the QA skill says.

## Insight 10: "The Models Will Fix This" is a Deferral, Not an Answer

**The argument:** Future models will catch the issues current models miss. Better AI will make these quality failures obsolete.

**The problem with this argument:**
1. It doesn't help you right now. The broken production site is live now. The missing alt tags exist now.
2. It assumes model improvement addresses comprehension gaps — but if the author doesn't know to specify a quality criterion, the model can't infer it
3. It creates a permanent present-tense deferral: there will always be a "next model" that will fix what the current one can't
4. It positions quality as a model responsibility rather than a developer responsibility — which is the root of the problem

**The deeper issue:** Even if future models catch everything a human expert would catch, that doesn't eliminate the need for the developer to understand what's in their codebase. Understanding is not only about catching bugs — it's about being able to reason about the system, extend it intentionally, debug unexpected failures, and make architectural decisions. A model that generates perfect code you don't understand doesn't give you those capabilities.

## Summary: The Four Failure Modes

| Failure Mode | Mechanism | Symptom | Countermeasure |
|---|---|---|---|
| **Metric displacement** | LOC / streak replaces quality signal | Bad code shipped confidently | Measure review depth, defect density |
| **Comprehension bypass** | AI generates, author ships without reading | Production failures that basics would catch | Read every line; set review-capacity limits |
| **Local optimization** | AI solves problems at nearest scope | Codebase complexity accumulates invisibly | Static analysis; architectural review gates |
| **Validation theater** | AI QA skills and automated scores replace real review | False confidence; known failures undiscovered | Deterministic tools; human comprehension gates |

---

## What Responsible AI-Assisted Development Looks Like

1. **Own comprehension.** You must be able to explain your system. If you can't, you haven't finished building it.
2. **Set a review budget.** Generate no more code per day than you can read and understand. Generation speed is not the constraint.
3. **Use deterministic validation.** Static analysis, linters, bundle analyzers, accessibility checkers. These are not optional.
4. **Separate generation from validation.** Don't use the same AI that generated the code to validate it. Use a different model, a different prompt, or a deterministic tool.
5. **Define "done" precisely.** "No bugs" is not a spec. Bundle size limits, accessibility compliance, test coverage thresholds, image optimization requirements — these are specs.
6. **Audit your production artifacts.** Look at what you're actually shipping. Open the network tab. Check the bundle. Run Lighthouse and then look at what Lighthouse doesn't check.
7. **Be skeptical of velocity metrics.** Shipping streaks, LOC counts, and "built in 2 hours" claims tell you nothing about quality. They are content, not evidence.
8. **Distrust advice proportional to its confidence.** The people most confident about AI coding practices are often the least equipped to evaluate the quality of what they've shipped.
